# BoostronixX theme

Custom WordPress theme converted from the static HTML designs in `D:\html\boostronixx`.
Presentation layer only — site logic (CPTs, etc.) belongs in a `project-core` plugin.

## Structure

```
boostronixx/
├── style.css              theme header (styling lives in compiled CSS)
├── functions.php          constants + includes + small template helpers
├── inc/
│   ├── theme-support.php  add_theme_support, nav menus
│   ├── enqueue.php        fonts (Bricolage always; Caveat→schools, JetBrains→hospitals),
│   │                      compiled CSS, Iconify, app.js (+ window.BX img/url injection)
│   └── dependency-guard.php  ACF/Yoast guard (admin notice, no fatal) + bx_field() helper
├── header.php             <head> (Yoast owns SEO meta), top nav, mobile drawer
├── footer.php             footer, mobile appnav (active state in PHP), lead-popup include
├── template-parts/lead-popup.php   home-only lead modal markup
├── front-page.php         home
├── page-{slug}.php        one per design page (WP auto-matches by page slug)
├── 404.php                designed not-found page
├── page.php / index.php   generic fallbacks
└── assets/
    ├── css/src/input.css  Tailwind v4 input: @import + @theme/@utility + custom CSS union
    ├── css/app.min.css    COMPILED output (committed)
    ├── js/app.js          all behaviours (guarded; site-wide safe)
    └── img/               local images copied from the design folder
```

## Rebuilding the CSS (Tailwind v4)

Styling is compiled, not run from a CDN (performance). To rebuild after editing
`assets/css/src/input.css` or after adding classes in templates/JS:

1. Get the Tailwind v4 **standalone CLI** (no Node project needed):
   `tailwindcss-windows-x64.exe` from https://github.com/tailwindlabs/tailwindcss/releases/latest
2. From this theme directory, run:

   ```
   tailwindcss -i assets/css/src/input.css -o assets/css/app.min.css --minify
   ```

`@source "../../../"` in `input.css` makes Tailwind scan every PHP template **and** `app.js`
(some utility classes are generated in JS), so all used classes are compiled.

### CSS scoping (important)
The source pages reuse generic class names (`.field`, `.chip`, `.orb`, `.prose`, …) with
DIFFERENT styling per page. To stop them bleeding across pages, `input.css` is structured as:
- **shared base** (from `index.html`) — global.
- **per-page CSS** — every other page's rules are scoped under `.bx-page-{slug}` (added to
  `<body>` by the `bx_body_class` filter in `inc/theme-support.php`; 404 gets `bx-page-404`).
- **colliding `@keyframes`** (same name, different motion on different pages) are namespaced
  per page (`name_slug`).

So when editing a single page's look, edit the rules under its `/* ---- {slug} ---- */`
block. `input.css` is generated by a build script (`build_css.py`, kept in the session
scratchpad) from the source HTML — but it is now the maintained source of truth; edit it
directly and recompile.
